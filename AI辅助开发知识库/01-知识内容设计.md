# 知识内容设计

知识库的质量，决定了 AI 回答的上限。这篇文档解决一个核心问题：**该往知识库里放什么**。

---

## 判断标准

一条内容值不值得放进知识库，用三个问题来判断：

1. **稳定吗？** — 这条知识近期会频繁变动吗？频繁变动的内容维护成本极高，且容易过时变成错误知识。
2. **隐性吗？** — 阅读代码能直接理解这条知识吗？能直接看出来的不需要重复存储。
3. **复用价值高吗？** — 这条知识在多个开发场景下都会用到吗？只用一次的知识不值得入库。

---

## 应该存什么

### 架构决策

架构决策是知识库最核心的内容。记录的重点不是"选了什么"，而是"为什么选"和"有什么约束"。

一个好的架构决策文档模板：

```markdown
---
category: architecture
domain: system
priority: high
tags: [cqrs, mediatr]
last_verified: 2026-03-01
---

# 为什么采用 CQRS 模式

## 背景
系统读写操作在性能特征、一致性要求上差异显著，
统一模型难以兼顾两侧的优化需求。

## 决策
读写分离：Command 走完整领域逻辑，Query 直接查库返回 DTO。

## 约束
- Command Handler 内不允许直接返回查询结果
- Query 不得修改任何状态
- 新增功能前先判断属于 Command 还是 Query

## 权衡
代码量增加（每个操作都要 Command/Query + Handler），
换取读写独立优化的能力。
```

### 业务规则

业务约束是最容易被 AI 遗漏的部分，也是最容易产生错误建议的来源。

```markdown
---
category: rule
domain: order
priority: high
---

# 订单状态转换规则

## 规则
订单允许任意状态之间转换，不存在状态机约束。

## 原因
运营人员有手动纠错的需求，强制状态机会阻碍问题修复。

## 常见错误
❌ 不要添加"只有待支付才能取消"之类的校验
❌ 不要在状态切换时抛出 InvalidOperationException
```

### 领域术语

消除 AI 对项目专有名词的歧义理解，尤其是与通用概念同名但含义不同的术语。

```markdown
---
category: term
domain: system
---

# 领域术语表

## Task（任务）
核心业务实体，一个订单可包含多个 Task。
每个 Task 对应一个具体的可独立执行的工作单元。
注意：不是操作系统意义上的"任务"。

## Item（已废弃）
旧版本中的子实体概念，现已合并进 Task。
代码中如出现 Item 相关逻辑，均属遗留代码，不要参考。
```

### 接口契约

API 的格式约定、错误码语义、版本策略等，是 AI 生成接口代码时最容易出错的地方。

```markdown
---
category: contract
domain: api
---

# API 错误响应格式

## 统一结构
{
  "type": "错误类型标识",
  "message": "用户可读的描述",
  "errors": {}  // 可选，字段级别的详细错误
}

## 约定
- 不使用 200 状态码包装错误
- 5xx 错误不暴露内部堆栈信息
- 403 不返回详情（防止信息枚举）
```

### 已知陷阱

踩过的坑，记录下来可以避免重复踩，也可以让 AI 在遇到相关场景时主动提醒。

```markdown
---
category: pitfall
domain: database
---

# SQLite 不支持服务端 DateTimeOffset 排序

## 问题
ORM 生成的 ORDER BY 子句在 SQLite 上处理 DateTimeOffset 时，
结果可能不正确，部分版本甚至直接报错。

## 解决方案
先将数据拉到内存，再在客户端排序：
.AsEnumerable().OrderByDescending(x => x.CreatedAt)

## 影响范围
仅 SQLite Provider，生产数据库（PostgreSQL 等）不受影响。
使用内存数据库跑单元测试时需特别注意。
```

## 不应该存什么

| 内容类型 | 原因 |
|---------|------|
| 完整源代码文件 | 频繁变动，极难保持同步 |
| 数据库 DDL | 由迁移文件管理，重复且易过时 |
| 详细实现步骤 | 和具体代码耦合，随代码变化失效 |
| 会议纪要 | 难以检索，大量噪音 |
| 临时修复记录 | 不稳定，污染知识库 |

## Chunk 策略

### 原则：一个文件，一个完整知识点

```
❌ 把一篇长文档按固定字数切块
   → 一个知识点可能被切断，检索结果残缺

✅ 每个知识点独立一个文件，文件就是 Chunk
   → 检索到的每条结果都是完整可用的
```

### 文件大小参考

- **最小**：能独立表达一个完整概念（约 100 Token）
- **最大**：不超过 800 Token（约 600 汉字 / 800 英文词）
- **超出时**：按子主题拆分为多个文件，通过 Wiki 链接互相关联

### Frontmatter 元数据

每个知识文件建议使用统一的元数据格式：

```yaml
---
category: architecture | rule | term | pitfall | contract
domain: 业务域（如 order, user, payment, system）
priority: high | medium | low
tags: [相关技术标签]
last_verified: YYYY-MM-DD
---
```

`last_verified` 用于定期审查。超过设定周期未确认的条目需要人工复核，防止知识过时。

---

## 目录结构建议

按类别组织，而非按业务域。类别决定了检索时如何过滤，业务域通过元数据表达。

```
knowledge/
├── architecture/     # 架构决策
├── rules/            # 业务规则
├── terms/            # 领域术语
├── contracts/        # 接口契约
└── pitfalls/         # 已知陷阱
```
