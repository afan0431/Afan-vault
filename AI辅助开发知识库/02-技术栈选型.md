# 技术栈选型

这篇文档解释支撑知识库系统的各个技术组件是什么、各自解决什么问题，以及怎么在不同选项之间做选择。

---

## 整体组件图

```
知识文档（Markdown）
      ↓ 建库时（离线）
Embedding 模型  →  向量数据库（Qdrant）
      ↓ 查询时（实时）
Embedding 模型  →  向量搜索  →  Reranker  →  LLM
```

每个组件职责清晰、可独立替换。

---

## Embedding 模型

### 它是做什么的

Embedding 模型把文本转换成一组浮点数（向量），语义相近的文本转换出的向量在数学空间中距离近。这是"语义搜索"的基础——通过向量距离来衡量文本的相似程度。

```
"订单可以随意改状态"  →  [0.2, 0.8, 0.1, ...]
"状态转换无限制"      →  [0.2, 0.7, 0.1, ...]  ← 距离近，相关
"用户密码长度要求"    →  [0.9, 0.1, 0.6, ...]  ← 距离远，不相关
```

### 选型

**BGE-M3**（开源，自托管）
- 支持中英文，效果均衡
- 支持 Dense（语义）和 Sparse（关键词）双模式，天然支持混合检索
- 可通过 Ollama 在本地运行，无调用成本
- 适合：离线建库、对成本敏感、有本地 GPU 的场景

**OpenAI text-embedding-3-small**（托管 API）
- 质量稳定，接入简单
- 按 Token 计费，价格极低（$0.02/百万 Token，知识库全量建库通常不到 $0.01）
- 适合：无本地 GPU、希望快速起步、不想维护本地模型服务的场景

### 建库和查询使用同一模型

这一点必须注意：**建库时用什么模型生成向量，查询时必须用同一个模型**。不同模型的向量空间不兼容，混用会导致检索结果完全错误。

如果建库用 BGE-M3（本地），查询时可以继续用 BGE-M3，也可以用 OpenAI API——但前提是确认两者的向量维度相同、语义空间对齐，实践中通常不混用。

## 向量数据库

### 它是做什么的

普通数据库擅长精确匹配（WHERE id = 1），不擅长"找最相近的"。向量数据库专为最近邻搜索设计，能在数百万条向量中，毫秒级找出与查询向量最接近的 N 条。

### Qdrant（推荐）

Rust 实现，性能好，原生支持：
- **混合检索**：向量（语义）+ 稀疏向量（关键词）同时搜索，互补召回
- **Payload 过滤**：可以按元数据字段过滤（如只搜 `category=rule` 的条目）
- Docker 单节点即可运行，资源占用极低（50MB 内存起步）

```bash
docker run -d -p 6333:6333 qdrant/qdrant
```

### pgvector（备选）

PostgreSQL 的向量扩展插件。如果项目已经在用 PostgreSQL，加一个插件即可支持向量存储，无需引入新的基础设施。功能上比 Qdrant 简单，但对于中小规模知识库（万条以内）完全够用。

### Chroma（本地开发验证）

Python 原生，零配置，几行代码就能跑起来。适合在正式选型前快速验证思路，不建议用于生产。

## Reranker（精排模型）

### 它是做什么的

向量检索速度快，但精度有限——因为查询向量和文档向量是分别独立计算的，无法捕捉两者之间细粒度的语义交叉。

Reranker（Cross-Encoder）的工作方式不同：把查询和候选文档**拼在一起**输入模型，让模型同时"读"两者，深度判断相关性。精度更高，但计算量大，只适合处理少量候选。

### 标准两阶段方案

```
第一阶段：向量检索（快）
  从全量知识库中，毫秒级检索 Top 50 候选

第二阶段：Reranker 精排（准）
  对 50 个候选重新打分，取 Top 5 给 LLM

为什么不直接用 Reranker 全量扫描？
→ Reranker 每次要读完整文档对，全量扫描会慢几个数量级。
```

### 是否必须加

知识库条目较少（< 500 条）时，向量检索的精度通常已经够用，可以先不加 Reranker，后期根据实际效果决定是否补充。

推荐模型：**BGE-Reranker-v2-m3**（开源，中文友好，可通过 Ollama 本地运行）

## 部署拓扑参考

### 最简配置（单机本地）

```
开发机
  ├── Ollama（运行 BGE-M3）
  └── Qdrant（Docker）
```

适合个人开发、知识库规模小的场景。

### 分离部署（推荐生产）

```
开发机（高性能 GPU）
  └── Ollama（离线建库用，不需要常驻）

内网服务器（低功耗长期运行）
  └── Qdrant（常驻，提供检索服务）

查询时 Embedding → OpenAI API（按需调用）
```

建库时用本地 GPU 跑 BGE-M3，查询时用 OpenAI API——前者性能好适合批量，后者无需常驻 GPU 服务适合低频实时调用。
