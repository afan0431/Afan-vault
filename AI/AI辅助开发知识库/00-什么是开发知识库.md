# 什么是 AI 辅助开发知识库

## 背景

使用 AI 编程助手时，最常见的问题不是模型不够聪明，而是模型对你的项目一无所知。每次对话都需要手动解释背景，或者把大量代码粘贴进去，既浪费 Token，效果也参差不齐。

一个典型的场景：你让 AI 帮你实现某个功能，它给出了看起来合理的方案，但那个方案违反了项目早就确定的架构约束，或者使用了已经被废弃的模式。问题不在于 AI 不会，而在于它不知道。

**AI 辅助开发知识库**的目标，就是把这些"只有长期参与项目的人才知道"的隐性知识结构化地存下来，让 AI 在回答问题前先查阅它。

---

## 核心思路：RAG

这套系统的底层技术是 **RAG（Retrieval-Augmented Generation，检索增强生成）**，分三步：

1. **建库**：把知识文档向量化，存入向量数据库
2. **检索**：提问时，先从知识库找出最相关的几条
3. **生成**：把检索到的知识和问题一起发给 LLM，得到有依据的回答

```
┌─────────────────────────────────────────┐
│         知识源（Markdown 文档）           │
│  架构决策 / 业务规则 / 术语 / 已知陷阱    │
└──────────────────┬──────────────────────┘
                   │ 离线批量处理
                   ▼
┌─────────────────────────────────────────┐
│            Embedding 管道               │
│    文本 → 向量（浮点数数组） → 向量库    │
└──────────────────┬──────────────────────┘
                   │ 实时检索
                   ▼
┌─────────────────────────────────────────┐
│            AI 查询流程                   │
│  提问 → 检索 Top5 → 注入 Prompt → LLM   │
└─────────────────────────────────────────┘
```

---

## 解决什么问题

| 问题 | 没有知识库 | 有知识库 |
|------|-----------|---------|
| 上下文传递 | 手动粘贴代码和说明，每次 ~5000 Token | 自动检索相关片段，~500 Token |
| 知识准确性 | AI 凭训练记忆猜测项目规则，常出偏差 | AI 读取精确的规则描述 |
| 建议一致性 | 不同对话中 AI 给出矛盾建议 | 统一知识源，建议稳定 |
| 规则更新 | 规则变了要在每次对话中反复说明 | 更新知识库文件，全局生效 |

---

## 与直接提供代码上下文的区别

很多工具会自动将相关代码文件注入上下文，这是合理的——但两种方式定位不同：

- **代码上下文**：告诉 AI「现在的代码是什么样的」
- **知识库上下文**：告诉 AI「为什么这样设计，有哪些约束」

前者描述现状，后者解释意图。两者配合效果最好。如果只能选一个起步，知识库在减少重复解释、保持建议一致性上的价值更高。

---

## 一个重要原则

> **知识库存的是"为什么"，不是"是什么"**
>
> 代码本身已经描述了系统的现状。知识库要补充的是背景、决策依据、业务约束——这些在代码里通常看不出来。

---

## 适用场景

比较适合：
- 有一定积累的长期项目（有架构决策和业务规则沉淀）
- 业务规则复杂，规则不体现在代码命名中的系统
- 团队协作场景（统一 AI 获取到的项目认知）

不太适合：
- 新项目或原型验证阶段（知识还在快速变化，维护成本高）
- 纯技术类项目（规则简单，代码即文档）
