# 维护工作流

> 知识库的价值取决于它的时效性。
> 过时的知识比没有知识更危险——AI 会基于错误信息给出错误建议。

知识库不是建好就完成了，维护才是长期运转的关键。这篇文档覆盖三个维度：**触发更新的时机**、**定期审查机制**、**知识库增长策略**。

---

## 触发更新的时机

| 触发事件 | 动作 |
|---------|------|
| 架构决策变更 | 更新对应文档，重新入库 |
| 业务规则修改 | 更新规则文件，更新 `last_verified` |
| 发现新的坑点 | 新建 pitfall 文件，入库 |
| 接口契约变更 | 更新 contract 文件，入库 |
| 旧决策被推翻 | 删除或标记废弃，从 Qdrant 删除对应条目 |

**原则：知识变更和代码变更同步提交。** 把知识文件纳入 Git 管理，和代码在同一个仓库（或独立的知识仓库），确保两者的变更历史可以对应。

---

## 增量入库

知识有变更时，无需全量重建，只处理变更的文件：

```bash
# 找出相对上次提交变更的知识文件
git diff --name-only HEAD | grep "^knowledge/.*\.md$" | xargs python index_knowledge.py --files
```

对于个人使用场景，知识库规模通常不大（百条级别），全量重建也只需几十秒，直接跑 `python index_knowledge.py` 是最省心的方式。

## 定期审查机制

### 为什么需要定期审查

知识文件不会自动跟随代码变化。一个三个月前写的架构决策描述，可能已经被一次重构悄悄推翻了，但知识库里还是旧的版本。

解决方案是给每条知识设置 `last_verified` 日期，定期检查哪些条目超过了验证周期。

### 自动检测过期知识

```python
# audit_knowledge.py
from pathlib import Path
from datetime import date
import yaml

KNOWLEDGE_DIR = "./knowledge"
STALE_DAYS = 90  # 超过此天数未确认则视为待审查


def audit():
    today = date.today()
    stale = []

    for path in Path(KNOWLEDGE_DIR).rglob("*.md"):
        content = path.read_text(encoding="utf-8")
        if not content.startswith("---"):
            continue

        end = content.find("---", 3)
        if end == -1:
            continue

        try:
            fm = yaml.safe_load(content[3:end])
        except yaml.YAMLError:
            continue

        last_verified = fm.get("last_verified")
        if not last_verified:
            stale.append(("未设置", path.name, "—"))
            continue

        delta = (today - date.fromisoformat(str(last_verified))).days
        if delta > STALE_DAYS:
            stale.append((delta, path.name, str(last_verified)))

    if not stale:
        print(f"所有知识均在 {STALE_DAYS} 天内已验证 ✓")
        return

    stale.sort(key=lambda x: x[0] if isinstance(x[0], int) else 9999, reverse=True)
    print(f"待审查条目：{len(stale)} 条\n")
    for days, name, verified in stale:
        label = f"{days} 天前" if isinstance(days, int) else days
        print(f"  {label:>10}  {name}  （最后验证：{verified}）")


if __name__ == "__main__":
    audit()
```

### 审查动作

对每个过期条目：

- **内容仍然准确** → 只更新 `last_verified` 日期，重新入库
- **需要更新** → 修改内容，更新日期，重新入库
- **已经过时** → 删除文件，并从 Qdrant 删除对应向量

```python
# 从 Qdrant 删除指定文件的向量
import hashlib, httpx

def delete_entry(file_path: str):
    doc_id = str(int(hashlib.md5(file_path.encode()).hexdigest(), 16) % (10**15))
    httpx.post(
        f"{QDRANT_URL}/collections/{COLLECTION}/points/delete",
        json={"points": [doc_id]},
    )
```

## 知识库增长策略

### 不要追求数量

200 条高质量、准确的知识 > 2000 条低质量、过时的知识。

知识库条目越多，噪音越多，检索精度越低，维护负担越重。只有经过沉淀、确认有复用价值的知识才值得入库。

### 推荐的增长节奏

```
项目初期（建库第一周）
  └── 建立基础骨架，20-30 条
      ├── 核心架构决策（5 条）
      ├── 关键业务规则（10 条）
      ├── 主要领域术语（5 条）
      └── 已知陷阱（5-10 条）

日常开发阶段
  └── 遇到值得记录的知识点，当场写进去
      原则：当你需要向新同事解释某件事时，就是它该进知识库的时候

定期整理（每月）
  └── 运行 audit.py，处理过期条目
      删除不再适用的内容，补充遗漏的重要知识
```

### 什么时候知识"该进库了"

一个实用的判断标准：

> 当你向 AI 解释了同一件事超过两次，这件事就应该进知识库。

## 用 Git 管理知识源

将 `knowledge/` 目录纳入 Git 是强烈推荐的做法。

好处：
1. **历史可追溯** — 知道每条知识是何时添加、为什么修改的
2. **增量更新** — `git diff` 精确找出变更文件，只处理需要重新入库的部分
3. **可回滚** — 知识写错了，可以 revert

```
knowledge/
├── .gitignore        # 不追踪 __pycache__ 等
├── architecture/
├── rules/
├── terms/
├── contracts/
└── pitfalls/
```

知识文件变更时，建议和代码变更分开提交，方便区分"功能变更"和"知识沉淀"两类提交。
